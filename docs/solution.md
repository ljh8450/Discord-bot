# 개발자 기회 레이더 — 문제·솔루션 적합성 평가 및 최종안

## 1. 결론

현재 아이디어는 사용자 페인 포인트를 해결할 수 있다. 다만 모든 정보 유형을 동일한 방식으로 실시간 전송하고, 사용자별 설정과 검색 기능까지 첫 버전에 넣으면 니치함과 운영 집중도가 무너진다.

최종 솔루션은 다음 두 기능으로 제한한다.

1. **Opportunity Radar**: 수도권 신입 개발자가 실제로 지원할 수 있는 채용·해커톤·공모전·교육 활동을 발견 후 빠르게 Discord에 알린다.
2. **Developer Brief**: AI-native 개발과 개발 산업 변화를 이해하는 데 도움이 되는 콘텐츠를 하나의 Discord 채널에 하루 한 번 큐레이팅한다.

두 기능은 수집·중복 제거·근거 검증 파이프라인을 공유하지만, 평가 기준과 발송 방식을 분리한다. 제품의 핵심은 Opportunity Radar이며 Developer Brief는 보조 기능이다.

최종 루브릭 점수는 **88/100**이다. 80점 기준을 통과했다.

## 2. 해결할 문제

### 핵심 사용자

- 대학교 졸업 예정이며 취업 준비를 시작한 신입 개발자
- 친구 몇 명과 개인 Discord 서버를 함께 사용
- 수도권의 신입·인턴 기회를 찾고 있음
- 선호 직무는 백엔드, 풀스택, FDE, 프론트엔드, AI Engineer
- 기술 스택 자체보다 직무와 회사의 장기적인 커리어 가치를 중요하게 봄

### 핵심 페인 포인트

사용자는 채용·해커톤·공모전·교육 정보를 찾기 위해 여러 사이트와 채널을 매일 순회해야 한다. 각 사이트에서도 경력, 직무, 지역, 활동 성격을 직접 읽고 걸러야 하며 오래된 공고 사이에서 새 공고를 찾아야 한다.

그 결과 다음 손해가 발생한다.

- 모집 기간이 짧은 해커톤과 공모전을 놓친다.
- 공고를 늦게 발견해 기업 조사, 포트폴리오 정리, 팀 구성 시간이 줄어든다.
- 경력직과 비개발 활동 알림이 많아 중요한 알림까지 무시하게 된다.
- 사이트별 큐레이션 채널을 여러 개 구독해도 전체 출처를 포괄하지 못한다.
- AI와 개발 방식의 변화를 따라가기 위해 별도의 블로그·유튜브·뉴스·커뮤니티를 다시 순회한다.

### 문제 문장

> 수도권 신입 개발자 취업 준비생은 자신이 지원할 수 있는 개발 기회가 여러 출처에 분산되어 있어 반복 탐색과 수동 필터링에 시간을 쓰고, 중요한 공고를 늦게 발견해 준비 경쟁에서 불리해진다.

### 사용자 핵심 과업

> 내가 지원할 만한 기회를 놓치지 않고 일찍 발견해, 다른 사람보다 빠르게 조사와 지원 준비를 시작하고 싶다.

콘텐츠 소비는 독립된 핵심 과업이 아니라 다음 보조 과업으로 둔다.

> 여러 채널을 찾아다니지 않고도 AI 시대의 개발 방식과 산업 변화를 하루 한 번 따라가고 싶다.

## 6. 최종 솔루션

### Discord 정보 구조

초기에는 정확히 두 채널만 운영한다.

#### `#opportunities`

사용자가 실제로 지원·신청해야 하는 기회를 개별 메시지로 빠르게 전달한다.

- 채용·인턴
- 개발 해커톤·공모전
- 개발 동아리
- 기업 교육
- 개발 관련 멘토링
- 혜택이 우수한 기타 교육·대외활동

#### `#dev-feed`

읽을거리를 하루 한 번 묶어서 전달한다.

- 기술 블로그
- 선별된 개발 YouTube
- GitHub Trending
- 주요 오픈소스 릴리스
- AI 뉴스
- 개발 커뮤니티 인기 글

콘텐츠 때문에 `#opportunities`의 긴급 알림이 묻히지 않도록 두 채널을 합치지 않는다. “한 채널에서 큐레이팅”한다는 요구는 모든 콘텐츠를 `#dev-feed` 하나에 모으는 것으로 충족한다.

### 기능 1 — 신규 기회 자동 발견

- 등록된 출처를 GitHub Actions로 주기적으로 확인한다.
- 새 항목과 기존 항목의 중요 변경을 감지한다.
- 동일 공고의 URL이 여러 개면 최초 발견 링크를 사용자에게 제공한다.
- 공식 원문이 따로 있다면 내부 검증 근거로 함께 저장한다.
- 한 번 보낸 동일 이벤트는 다시 보내지 않는다.

### 기능 2 — 공통 사용자 프로필 기반 필터

첫 버전에는 사용자별 설정 UI를 만들지 않는다. 저장소 설정 파일에 서버 공통 프로필 한 개를 둔다.

#### 채용 하드 필터

- 신입 또는 인턴 지원 가능
- 수도권 근무: 서울·경기·인천

#### 채용 선호 직무

- 백엔드
- 풀스택
- FDE
- 프론트엔드
- AI Engineer

#### 보수적 노출

- C 중심 직무
- 임베디드
- .NET 중심 직무
- 커리어 확장성이 제한적이라고 판단되는 직무

단, 대기업 또는 유니콘 기업은 기술 스택과 관계없이 통과시킨다. 대기업·유니콘 목록은 설명 가능한 명시적 allowlist로 관리한다.

#### 해커톤·공모전

- 개발 결과물을 만드는 행사이면 통과
- 참가 형태, 지역, 학생 여부는 하드 필터로 제한하지 않음

#### 교육·대외활동

바로 통과:

- 개발 동아리
- 기업 주관 개발 교육
- 개발 관련 멘토링

추가 검증:

- 유료 프로그램
- 장기간 풀타임 과정
- 부트캠프
- 정부 지원 교육
- 창업·서포터즈 프로그램

### 기능 3 — 최소 검증과 `pending` 혜택 심사

일반 기회는 다음 검증을 한 번 통과하면 빠르게 발송한다.

1. 원문 또는 신뢰할 수 있는 최초 발견 페이지가 열리는가
2. 현재 모집 중인가
3. 신입·인턴 또는 대상 사용자가 지원 가능한가
4. 이전에 발송한 대상과 중복이 아닌가
5. 한 줄 요약이 본문 또는 기업 공식 자료에 근거하는가

추가 판단이 필요한 교육·활동은 `pending`으로 저장하고 다음 실행에서 혜택을 평가한다.

```text
DISCOVERED -> VALIDATING -> APPROVED -> SENT
                       \-> PENDING_BENEFIT -> APPROVED | REJECTED
                       \-> REJECTED
```

혜택 검증 요소:

- 무료 또는 비용 지원
- 활동비·장학금·장비 지원
- 기업 인지도와 주최 기관 신뢰도
- 현직자 멘토링
- 실제 프로젝트와 포트폴리오 결과물
- 채용 연계
- 수료 인증
- 기간과 시간 부담

### 기능 4 — 빠른 알림과 일일 안전망

- 기회는 검증 통과 후 가능한 한 즉시 발송한다.
- GitHub Actions의 실제 의미는 실시간 푸시가 아니라 짧은 주기의 best-effort polling이다.
- 파일럿 목표는 공식 출처 게시 후 30분 이내 탐지다.
- 매일 한 번 최근 24시간 항목을 재검사해 실행 실패나 일시적인 파싱 오류로 놓친 항목을 복구한다.
- 알림 수 상한은 초기에는 두지 않는다. 비용이나 Discord rate limit 문제가 실제로 관찰될 때 도입한다.

### 기능 5 — 컴팩트한 Discord 알림

```text
🚀 [신입 채용] 네이버 백엔드 개발자 모집

신입 지원 가능 · 서울 · 7월 31일 마감
한 줄: 커머스 주문 플랫폼을 개발하는 백엔드 포지션

[공고 보기] [관심 있어요] [지원했어요] [관련 없어요]
```

필수 정보는 제목과 링크다. 지역·지원 가능 여부·마감일·근거 있는 한 줄 요약만 추가한다. 긴 AI 요약과 추천 점수는 노출하지 않는다.

### 기능 6 — 일일 Developer Brief

`#dev-feed`에는 하루 한 번만 발송한다. 주요 보안 릴리스처럼 즉시성이 명확한 항목만 예외적으로 개별 전송할 수 있다.

선별 점수:

```text
content_score = relevance + practical_value + freshness + authority + novelty
```

- AI-native 개발, Agent, MCP, 백엔드, 프론트엔드, 개발자 커리어와 관련 있는가
- 실제 구현·현업 사례·원문 근거가 있는가
- 기존에 보낸 콘텐츠와 새로운 차이가 있는가
- 단순 홍보·루머·중복 기사인가

YouTube 초기 핵심 출처:

- 실밸개발자
- AI Frontier Korea
- Tech Bridge
- 미드나잇 로그 Midnight Log
- 조코딩 JoCoding

선택적 출처:

- EO Korea: AI 스타트업·개발 조직 영상만
- 안될공학: AI 인프라·반도체·개발 도구 영상만

채널은 최근 활동성도 검사한다. 최근 90일 내 일반 영상이 없으면 휴면 처리하고, 새 영상이 감지되면 자동 복구한다.

### 기능 7 — 최소 행동 측정

메시지 버튼 세 개만 저장한다.

- `관심 있어요`
- `지원했어요`
- `관련 없어요`

초기에는 추천 학습에 사용하지 않고 제품 평가에만 사용한다. 사용자별 추천은 웹 확장 이후로 미룬다.

## 7. 핵심 지표와 가드레일

### North Star Metric

```text
지원 전환율 = 지원했어요 수 / 전달된 지원 가능 기회 수
```

이 지표는 자기보고 방식이므로 절대적인 실제 지원 완료율이 아니라 기간별·필터별 개선 추세를 보는 데 사용한다.

### 보조 지표

- 관심 반응률
- 관련 없음 비율
- 신규 게시부터 Discord 발송까지 걸린 시간
- 주요 공고 누락률
- 일일 Developer Brief 링크 클릭률

### 품질 가드레일

- 중복 알림 비율
- 마감·지원 자격 오류율
- 공식 근거 없는 요약 비율
- Discord 채널 음소거 또는 알림 비활성화
- `관련 없어요`가 연속해서 발생한 출처와 규칙

초기 판단 기준:

- 중복 알림 2% 미만
- 날짜·모집 상태 정확도 98% 이상
- 공식 출처 신규 항목 탐지 중앙값 30분 이내
- 관련 없음 반응 20% 이하

목표값은 2주 파일럿 후 실제 표본을 바탕으로 다시 정한다.

## 8. 서버리스 MVP 구조

```text
GitHub Actions Scheduler
  -> Source Adapters
  -> Raw Metadata
  -> Normalizer / Dedupe
  -> Rule Filter
  -> Validation / Benefit Pending
  -> State Store
  -> Discord Webhook
```

운영 원칙:

- 출처 어댑터와 Discord 발송을 직접 연결하지 않는다.
- 워크플로는 `concurrency: 1`로 직렬화해 중복 실행을 방지한다.
- 발송 전 `dedupeKey`를 생성하고 발송 완료 상태를 저장한다.
- Discord 발송 실패는 다음 실행에서 재시도한다.
- HTML 전체를 장기간 저장하지 않고 URL, 해시, 추출 근거와 필요한 메타데이터만 보관한다.

초기 상태는 소규모 파일 기반으로 시작할 수 있다. GitHub Actions 캐시는 영구 저장소로 보장되지 않으므로 기준 저장소로 사용하지 않는다. 파일 충돌이나 데이터 증가가 관찰되면 외부 서버리스 DB로 교체한다.

## 10. 검증 계획

친구들을 포함한 소규모 서버에서 2주간 운영한다.

검증할 가설:

> 공통 조건으로 강하게 필터링된 신규 기회 알림은 범용 채용 채널보다 실제 지원 전환율이 높고, 사용자가 알림을 귀찮아하지 않는다.

필수 시나리오:

1. 신규 지원 가능 공고를 한 번만 보낸다.
2. 경력직 또는 비수도권 공고를 보내지 않는다.
3. C·임베디드·.NET 공고는 보수적으로 제외하되 allowlist 기업은 통과한다.
4. 유료·장기 교육은 `pending` 혜택 검증 후에만 보낸다.
5. 한 줄 요약의 모든 주장이 원문 근거를 가진다.
6. 워크플로 실행 실패 후 일일 재검사에서 누락 항목을 복구한다.
7. 사용자가 메시지에서 관심·지원·관련 없음을 표시할 수 있다.
8. 콘텐츠는 기회 알림과 분리되어 하루 한 번만 발송된다.

### 2주 후 유지 결정

다음 조건을 함께 만족하면 출처를 확대한다.

- 핵심 사용자에게 실제 지원 사례가 발생함
- 관련 없음 반응이 20% 이하임
- 중복 알림이 2% 미만임
- 사용자가 `#opportunities` 알림을 끄지 않음
- 운영자가 매일 수동으로 고쳐야 하는 출처가 전체의 20% 미만임

충족하지 못하면 기능을 추가하지 않고 필터·출처·검증 규칙부터 수정한다.

## 11. 최종 평가

이 솔루션은 범용 개발 정보 봇이 아니다. 핵심 니치는 다음과 같다.

> 수도권 신입 개발자와 그 친구들이 지원 가능한 개발 기회를 남보다 빨리 발견하도록 돕는 Discord 레이더

콘텐츠 큐레이션은 별도 제품처럼 확장하지 않고, 동일 사용자가 개발·AI 변화를 따라가기 위한 하루 한 번의 보조 피드로 제한한다.

이 범위라면 사용자 페인 포인트와 기능 사이의 연결이 분명하고, GitHub Actions 기반 파일럿으로 검증할 수 있으며, 실패 시 어떤 필터나 출처를 고쳐야 하는지도 측정할 수 있다. 따라서 **88/100으로 구현 진행에 적합**하다고 판단한다.
