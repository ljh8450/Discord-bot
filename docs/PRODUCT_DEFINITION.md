# 개발자 기회 변화 감지 봇 — 제품 정의

## 1. 한 문장 정의

관심 분야와 조건에 맞는 **개발자 기회의 상태 변화**를 수집·판별해, 사용자가 행동해야 할 때 Discord로 알려주는 개인화 알림 봇이다.

이 제품의 핵심은 링크 수집이 아니다. 다음과 같은 변화(Change Event)를 신뢰성 있게 감지하는 것이다.

- 새로운 모집·신청·공고가 열림
- 접수 또는 지원 마감이 임박함
- 관심 기업이나 관심 기술에 관련된 기회가 생김
- 프로젝트·모델·콘텐츠가 새로 공개됨
- 관심도가 짧은 시간 안에 급상승함

## 2. 해결하려는 문제

개발자에게 유용한 기회는 기업 채용 페이지, 대외활동 사이트, 커뮤니티, 기술 블로그, GitHub 등 여러 곳에 흩어져 있다. 사용자는 이를 반복해서 확인하거나 여러 알림 채널을 구독해야 한다.

기존 정보 모음 방식에는 네 가지 문제가 있다.

1. **발견이 늦다.** 모집 시작이나 마감 직전에야 기회를 알게 된다.
2. **노이즈가 많다.** 새 글과 오래된 글, 실제 모집과 홍보 콘텐츠가 섞인다.
3. **개인화가 약하다.** 직무, 경력, 기술, 학생 지원 가능 여부처럼 실제 지원 판단에 필요한 조건이 반영되지 않는다.
4. **변화를 설명하지 않는다.** 무엇이 새로 생겼는지, 무엇이 바뀌었는지, 왜 지금 봐야 하는지가 드러나지 않는다.

따라서 해결할 문제는 다음과 같다.

> 사용자가 여러 사이트를 직접 순회하지 않아도, 자신에게 관련 있고 지금 행동할 가치가 있는 개발자 기회의 변화를 제때 알 수 있게 한다.

## 3. 핵심 사용자

### 1차 사용자

- 취업·인턴·대외활동을 준비하는 한국의 대학생 및 주니어 개발자
- 포트폴리오에 도움이 되는 활동을 찾는 사람
- 백엔드, 프론트엔드, AI 등 관심 분야가 비교적 명확한 사람

### 2차 사용자

- 이직 기회를 탐색하는 경력 개발자
- 특정 오픈소스, AI 모델, 기술 생태계의 변화를 추적하는 개발자
- 개발 커뮤니티 Discord 서버 운영자

초기 MVP는 1차 사용자에게 집중한다.

## 4. 사용자 핵심 과업

사용자가 달성하려는 일은 “개발 정보를 많이 읽는 것”이 아니다.

> 나에게 도움이 되는 기회를 늦지 않게 발견하고, 지원·신청 여부를 빠르게 결정하는 것.

이를 다음 흐름으로 지원한다.

1. 사용자가 관심 직무, 기술, 기업, 경력 수준을 설정한다.
2. 시스템이 등록된 출처를 주기적으로 확인한다.
3. 원문을 정규화하고 이전 상태와 비교한다.
4. 의미 있는 변화만 이벤트로 만든다.
5. 관련도, 긴급도, 신뢰도를 계산한다.
6. Discord에서 변화 이유와 다음 행동을 짧게 알린다.

## 5. 제품 원칙

### 변화가 없는 정보는 알리지 않는다

같은 공고를 반복 전송하지 않는다. URL뿐 아니라 원문의 식별자와 내용 지문을 이용해 중복을 판별한다.

### 행동 가능한 정보가 먼저다

마감일, 지원 자격, 모집 상태, 신청 링크처럼 사용자의 행동에 영향을 주는 정보가 우선이다.

### 원문과 근거를 남긴다

모든 알림은 원문 링크, 수집 시각, 출처를 포함한다. 추론한 값은 원문에 명시된 값과 구분한다.

### 확신이 낮으면 조용히 처리한다

날짜나 모집 상태를 확실하게 판별하지 못하면 잘못된 긴급 알림을 보내지 않는다. 검토 대기 상태로 두거나 일반 신규 알림으로 낮춘다.

### 개인화가 카테고리보다 중요하다

“채용” 전체를 구독하게 하기보다 `백엔드 + 신입 + 서울/원격 + Java/Spring`처럼 실제 선택 조건을 반영한다.

## 6. MVP 범위

초기 버전은 **기한이 있고 행동으로 이어지는 기회**에 집중한다.

| 우선순위 | 분야 | 감지 이벤트 | 주요 필터 |
|---|---|---|---|
| P0 | 채용·인턴 | 신규 등록, 마감 임박, 관심 기업 신규 채용 | 직무, 경력, 기술, 기업, 마감일 |
| P0 | 교육·커뮤니티 활동 | 부트캠프·동아리·프로그램 모집 시작, 마감 임박 | 학생 가능, 비용, 기간, 분야 |
| P0 | 해커톤·개발 대회 | 접수 시작, 마감 임박 | AI/Backend/Web, 학생 가능, 상금 |
| P1 | 컨퍼런스·세미나 | 신청 시작, 마감 임박 | 주제, 온라인/오프라인, 지역, 비용 |
| P1 | 지원사업·장학금 | 신규 모집, 마감 임박 | 대상, 지역, 지원 금액, 분야 |

다음은 수집 방식과 평가 기준이 달라 MVP 이후로 둔다.

| 후속 범위 | 감지 이벤트 | 뒤로 미루는 이유 |
|---|---|---|
| 기술 블로그·YouTube | 새 글·새 영상 | 행동 기한이 없어 초기 가치 검증 신호가 약함 |
| GitHub | Trending, Star 급상승, Release | 시계열 기준선과 별도 API 정책이 필요함 |
| AI 뉴스·모델 | 주요 뉴스, 모델 공개 | 중요도 판별과 중복 기사 군집화가 필요함 |
| 커뮤니티 인기글 | 일일 인기글 | 출처별 인기 기준과 품질 관리가 필요함 |

## 7. MVP에서 하지 않는 것

- 인터넷 전체를 대상으로 한 범용 크롤러
- AI가 원문 근거 없이 마감일이나 지원 자격을 추측하는 기능
- 모든 카테고리를 처음부터 동일 수준으로 지원하는 것
- Discord 메시지 안에서 지원서를 작성하거나 실제 지원을 대행하는 기능
- 원문 전체를 복제해 저장하거나 재배포하는 기능
- 완전한 추천 시스템: 초기에는 설명 가능한 규칙 기반 점수를 사용한다

## 8. 이벤트 모델

### Event와 Opportunity를 분리한다

`Opportunity`는 채용공고나 해커톤 같은 대상 자체이고, `ChangeEvent`는 그 대상에서 발생한 변화다.

예를 들어 하나의 채용공고는 생애주기 동안 여러 이벤트를 만든다.

```text
DISCOVERED -> OPENED -> DEADLINE_APPROACHING -> CLOSED
                  \-> UPDATED
```

### Opportunity 공통 필드

```json
{
  "id": "internal-id",
  "type": "JOB",
  "sourceId": "naver-careers",
  "externalId": "source-owned-id",
  "canonicalUrl": "https://...",
  "title": "신입 백엔드 개발자 채용",
  "organization": "네이버",
  "status": "OPEN",
  "publishedAt": "2026-07-20T00:00:00+09:00",
  "opensAt": "2026-07-20T09:00:00+09:00",
  "closesAt": "2026-07-31T18:00:00+09:00",
  "tags": ["backend", "entry-level", "java", "spring"],
  "eligibility": ["신입"],
  "contentHash": "sha256:...",
  "firstSeenAt": "2026-07-20T09:05:00+09:00",
  "lastSeenAt": "2026-07-20T09:05:00+09:00"
}
```

분야별 상세 값은 `attributes`에 둔다. 예를 들어 채용은 고용 형태와 경력, 해커톤은 상금과 팀 규모, 행사는 장소와 참가비를 가진다.

### ChangeEvent 공통 필드

```json
{
  "id": "event-id",
  "opportunityId": "internal-id",
  "eventType": "DEADLINE_APPROACHING",
  "detectedAt": "2026-07-28T09:00:00+09:00",
  "effectiveAt": "2026-07-28T09:00:00+09:00",
  "before": { "daysRemaining": 4 },
  "after": { "daysRemaining": 3 },
  "confidence": 0.98,
  "dedupeKey": "internal-id:deadline:d-3",
  "evidence": ["source-url"]
}
```

### 표준 이벤트 타입

- `DISCOVERED`: 처음 발견됨
- `OPENED`: 모집·지원·신청이 시작됨
- `UPDATED`: 지원 판단에 중요한 필드가 변경됨
- `DEADLINE_APPROACHING`: 정해진 기준일에 진입함(D-3, D-1, 당일)
- `CLOSED`: 모집 또는 신청이 종료됨
- `POPULARITY_SPIKE`: 관심도가 기준선보다 급상승함
- `RELEASED`: 새 버전 또는 모델이 공개됨
- `PUBLISHED`: 새 글이나 영상이 발행됨

`신규 채용`, `해커톤 모집 시작`, `세미나 신청 시작`은 서로 다른 내부 이벤트가 아니라 `Opportunity.type`과 표준 이벤트 타입의 조합으로 표현한다.

## 9. 변화 감지 규칙

### 새 항목

- 출처 고유 ID가 새롭거나 정규화 URL이 존재하지 않으면 후보가 된다.
- 제목·기관·마감일이 매우 유사하면 URL이 달라도 중복 후보로 묶는다.
- 최초 게시 시각을 확인할 수 없으면 `publishedAt`과 `firstSeenAt`을 구분한다.

### 변경

- 매 수집 결과의 정규화 필드를 이전 스냅샷과 비교한다.
- 제목 장식, 공백, 조회수처럼 중요하지 않은 차이는 무시한다.
- 마감일, 상태, 자격, 직무, 장소, 신청 URL 변경은 알림 가능한 변화로 본다.

### 마감 임박

- 기본 기준은 D-3, D-1, 당일이다.
- 같은 기준에 대해서는 한 번만 이벤트를 만든다.
- 종료 시각이 없는 날짜는 해당 지역 시간대의 23:59로 처리하되, 알림에 “시간 미표기”를 표시한다.
- 상시 모집과 채용 시 마감은 임박 알림 대상에서 제외한다.

### 관심 기업·개인화

- 별도 이벤트 타입을 만들지 않는다.
- 기존 이벤트에 사용자 관심 조건이 일치하면 우선순위와 알림 표시를 높인다.
- 따라서 “관심 기업 신규 채용”은 `JOB + DISCOVERED + company match`다.

## 10. 관련도와 알림 정책

초기 점수는 설명 가능한 규칙으로 계산한다.

```text
notification_score = relevance + urgency + source_trust + novelty - duplicate_risk
```

- 관련도: 관심 직무·기술·기업·경력 조건 일치
- 긴급도: 모집 시작, D-3, D-1, 오늘 마감
- 출처 신뢰도: 공식 사이트 > 공식 SNS/RSS > 검증된 집계 사이트 > 커뮤니티 제보
- 새로움: 최초 발견 또는 중요한 필드 변경
- 중복 위험: 동일 대상의 다른 URL, 재게시, 낮은 파싱 확신

사용자별 즉시 알림, 일일 요약, 알림 안 함 중 하나로 라우팅한다.

## 11. Discord 사용자 경험

### 최소 명령

- `/구독 설정`: 관심 분야, 직무, 기술, 기업, 경력 설정
- `/구독 보기`: 현재 조건과 알림 빈도 확인
- `/구독 끄기`: 일부 또는 전체 알림 중지
- `/기회 검색`: 현재 열려 있는 기회 조회
- `/마감 임박`: 내 조건에 맞는 D-3 이내 항목 조회
- `/피드백`: 유용함, 중복, 관련 없음, 정보 오류 표시

### 알림 한 건의 구성

```text
[D-3 · 관심 기업] 네이버 신입 백엔드 개발자 채용

왜 알림? 관심 기업 ‘네이버’, 직무 ‘백엔드’와 일치
마감: 7월 31일 18:00 (Asia/Seoul)
기술: Java, Spring
변화: 지원 마감 D-3 진입

[공고 보기]  [관심 없음]  [저장]
출처: 네이버 Careers · 확인 5분 전
```

알림은 “무엇”, “무슨 변화”, “왜 나에게 왔는지”, “언제까지 무엇을 해야 하는지”를 한 화면에서 설명해야 한다.

## 12. 성공 지표

초기에는 수집량보다 신뢰와 행동을 측정한다.

### 핵심 지표

- 주간 알림 수신 사용자 중 원문 클릭 비율
- 알림당 `유용함` 비율
- 사용자당 저장 또는 지원 의향 표시 수

### 품질 방어 지표

- 중복 알림 비율
- 잘못된 마감일·종료 상태 신고 비율
- 신규 항목 발견부터 알림까지 걸린 시간
- 알림 비활성화 및 서버 이탈 비율

초기 목표 예시는 중복 알림 2% 미만, 공식 출처 신규 항목 탐지 30분 이내, 날짜 필드 정확도 98% 이상이다. 실제 목표치는 파일럿 출처와 주기를 정한 뒤 확정한다.

## 13. 시스템 경계

```text
Source Adapter
  -> Raw Snapshot
  -> Normalizer / Classifier
  -> Opportunity Store
  -> Change Detector
  -> Event Store
  -> Matcher / Ranker
  -> Discord Notifier
```

- **Source Adapter**: API, RSS, 정적 HTML 등 출처별 수집
- **Raw Snapshot**: 파싱 오류를 재현할 수 있도록 원본 응답 메타데이터 보관
- **Normalizer**: 날짜, 기업, 태그, 상태를 공통 모델로 변환
- **Opportunity Store**: 대상의 최신 상태와 이전 스냅샷 관리
- **Change Detector**: 두 상태의 의미 있는 차이와 시간 기반 이벤트 생성
- **Matcher / Ranker**: 사용자 구독 조건과 알림 정책 적용
- **Notifier**: Discord 메시지 발송, 재시도, 전송 이력 관리

출처별 수집기와 Discord 전송 코드를 직접 연결하지 않는다. 중간에 정규화된 대상과 이벤트를 두어야 중복 제거, 재처리, 테스트가 가능하다.

## 14. MVP 검증 범위

첫 파일럿은 많은 출처보다 서로 다른 수집 방식을 대표하는 3~5개 공식 출처를 선택한다.

- 공식 채용 페이지 1~2개
- 구조가 안정적인 교육·활동 모집 출처 1개
- 공식 해커톤 또는 대회 출처 1개
- RSS/API가 있는 행사 출처 1개

검증 시나리오는 다음 다섯 가지면 충분하다.

1. 새 항목을 한 번만 알린다.
2. 같은 항목을 다시 수집해도 알리지 않는다.
3. 마감일이 변경되면 변경 내용을 알린다.
4. D-3, D-1, 당일 알림을 각각 한 번만 보낸다.
5. 사용자의 관심 조건과 맞지 않으면 즉시 알림을 보내지 않는다.

## 15. 구현 단계

### 1단계 — 도메인 코어

- Opportunity, Snapshot, ChangeEvent, Subscription 모델
- 정규화 URL과 중복 키
- 상태 비교 및 D-day 이벤트 생성
- 단위 테스트용 고정 시계

### 2단계 — 수집 파일럿

- 출처 어댑터 인터페이스
- 공식 출처 3~5개 구현
- 수집 실패, 파싱 실패, 변경 감지 관측 로그
- 재수집과 재처리 가능한 저장 구조

### 3단계 — Discord 전달

- 구독 설정 명령
- 개인화 매칭과 즉시/요약 알림
- 발송 멱등성, 실패 재시도, 피드백 버튼

### 4단계 — 운영 검증

- 1~2주간 소규모 파일럿
- 중복률, 오탐률, 클릭률 확인
- 사용자 피드백으로 필터와 우선순위 조정

### 5단계 — 후속 카테고리

- GitHub Trending/Release
- 기술 블로그/YouTube
- AI 모델 공개와 뉴스 군집화
- 커뮤니티 인기글

## 16. 구현 시작 전 결정할 항목

다음 결정은 기술 구현보다 제품 범위에 영향을 준다.

1. 개인 DM 중심인지, 서버 채널 피드 중심인지
2. 첫 사용자군이 대학생·취준생인지, 주니어 전체인지
3. 첫 3~5개 공식 출처
4. 즉시 알림과 일일 요약의 기본값
5. 운영 지역과 시간대: 초기 기본값은 대한민국/Asia/Seoul로 가정

## 17. 현재 권장안

- 대상: 한국의 대학생·취준생 및 주니어 개발자
- 전달: 개인별 조건은 DM, 공용 큐레이션은 서버 채널
- 초기 분야: 채용·인턴, 포트폴리오 활동, 해커톤·대회
- 알림: 관심 기업 신규 공고와 D-1은 즉시, 나머지는 일일 요약
- 출처: 공식 출처 우선, 집계 사이트는 발견 보조용
- 기술 접근: 규칙 기반 정규화·감지를 먼저 만들고 AI는 분류 보조에만 사용

이 범위에서 가장 먼저 검증할 가설은 다음과 같다.

> 사용자는 많은 링크보다, 자신의 조건에 맞고 왜 지금 확인해야 하는지가 설명된 모집 시작·마감 임박 알림을 더 유용하게 느낀다.
